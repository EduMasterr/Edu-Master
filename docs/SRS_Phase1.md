## 1. مقدمة (Introduction)

### 1.1 الغرض (Purpose)
الغرض من هذه الوثيقة هو تحديد متطلبات نظام **EduMaster Pro**، وهو نظام إدارة معاهد تعليمية متكامل (EMS) يدعم تعدد الفروع (Multi-Branch) والعمليات الهجينة (Hybrid: حضوري وأونلاين). يهدف النظام إلى أتمتة جميع العمليات الإدارية والتعليمية والمالية لضمان كفاءة التشغيل وتوفير تجربة مستخدم متميزة لجميع الأطراف المعنية.

### 1.2 نطاق المشروع (Scope)
نظام **EduMaster Pro** سيخدم شبكة من المعاهد التعليمية (مبدئياً 8 فروع) تغطي مجالات متنوعة (لغات، برمجة، أطفال، تدريب مهني). سيشمل النظام إدارة الطلاب، الموظفين، الكورسات، الحضور، المدفوعات، والتقارير عبر واجهة ويب مركزية.

### 1.3 التعريفات والاختصارات (Definitions & Abbreviations)
- **EMS:** Education Management System (نظام إدارة التعليم).
- **Super Admin:** المدير العام للنظام (صلاحيات كاملة على كل الفروع).
- **Branch Admin:** مدير الفرع (صلاحيات كاملة داخل الفرع فقط).
- **Hybrid:** نظام تعليم يجمع بين الحضور الفعلي والأونلاين.

## 2. الوصف العام (Overall Description)

### 2.1 منظور المنتج (Product Perspective)
النظام عبارة عن تطبيق ويب (Web Application) يعمل بنظام مركزي (Centralized Database) مع واجهات مخصصة لكل فرع. يتميز بتصميم عصري (Premium UI) ويدعم التوسع المستقبلي لعدد غير محدود من الفروع والطلاب.

### 2.2 خصائص المستخدمين (User Classes & Characteristics)
1.  **Super Admin:** الإدارة العليا، الوصول لكل البيانات، إعدادات النظام العالمية.
2.  **Branch Admin:** إدارة العمليات اليومية للفرع، متابعة الموظفين والطلاب في الفرع.
3.  **Teacher:** إدارة المحتوى التعليمي، رصد الحضور، إدخال الدرجات.
4.  **Employee (Reception/Sales):** تسجيل الطلاب، تحصيل الرسوم، متابعة الاستفسارات.
5.  **Student:** متابعة الجدول، الحضور، الدرجات، المواد التعليمية.
6.  **Parent:** متابعة أداء الأبناء، الحضور، والمدفوعات.

### 2.3 بيئة التشغيل (Operating Environment)
-   **Server:** Cloud-based (AWS/DigitalOcean/Azure) or Local Server.
-   **Database:** SQL (Standard Relational Database).
-   **Client:** متصفحات الويب الحديثة (Chrome, Firefox, Edge, Safari) ودعم الهواتف الذكية والأجهزة اللوحية.

## 3. المتطلبات الوظيفية (Functional Requirements)

### 3.1 وحدة المصادقة والصلاحيات (Authentication & Authorization)
-   نظام تسجيل دخول آمن لكل الأدوار.
-   نظام صلاحيات مبني على الأدوار (RBAC - Role Based Access Control).
-   دعم تعدد الجلسات وإدارة كلمات المرور.

### 3.2 وحدة إدارة الفروع (Multi-Branch Management)
-   إنشاء وتعديل بيانات الفروع (الاسم، العنوان، المدير).
-   لوحة تحكم مركزية لمراقبة أداء جميع الفروع.
-   تقارير مقارنة بين الفروع.

### 3.3 وحدة إدارة الطلاب (Student Management)
-   ملف إلكتروني شامل لكل طالب (بيانات شخصية، أكاديمية، مالية).
-   نظام تسجيل وقبول الطلاب في الكورسات والمجموعات.
-   تتبع حالة الطالب (نشط، منقطع، خريج).

### 3.4 وحدة الكورسات والمجموعات (Courses & Groups)
-   إدارة الكتالوج التعليمي (اللغات، البرمجة، إلخ).
-   إنشاء مجموعات دراسية وتعيين المدرسين والمواعيد.
-   دعم الجداول الدراسية المعقدة (Recurring Sessions).

### 3.5 وحدة الحضور والغياب (Attendance Automation)
-   تسجيل الحضور يومياً (يدوي أو عبر QR Code مستقبلاً).
-   تنبيهات تلقائية للغياب والتأخير (SMS/WhatsApp placeholders).
-   تقارير حضور تفصيلية لكل طالب ومجموعة.

### 3.6 وحدة الإدارة المالية (Financial Module)
-   إدارة الرسوم الدراسية وخطط الدفع (أقساط / كاش).
-   إصدار فواتير وسندات قبض.
-   تتبع المصروفات والإيرادات لكل فرع.
-   حساب رواتب الموظفين والمدرسين (بالساعة أو راتب ثابت).

## 4. المتطلبات غير الوظيفية (Non-Functional Requirements)

### 4.1 الأداء (Performance)
-   يجب أن يتحمل النظام 3000 طالب و 8 فروع دون بطء ملحوظ.
-   زمن استجابة الصفحات يجب أن يكون أقل من 2 ثانية في الظروف الطبيعية.

### 4.2 الأمان (Security)
-   تشفير بيانات كلمات المرور.
-   حماية ضد ثغرات الويب الشائعة (XSS, SQL Injection).
-   نسخ احتياطي دوري للبيانات.

### 4.3 التصميم وتجربة المستخدم (UI/UX)
-   واجهة عصرية تدعم الوضع الداكن (Dark Mode) بشكل افتراضي (مشابه للنظام الحالي).
-   دعم كامل للغة العربية (RTL).
-   تصميم متجاوب (Responsive) يعمل على جميع الأجهزة.

### 4.4 القابلية للتوسع (Scalability)
-   النظام مصمم ليكون Modular بحيث يمكن إضافة فروع أو موديولز جديدة دون إعادة هيكلة النظام.
